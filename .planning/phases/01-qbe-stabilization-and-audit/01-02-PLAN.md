Create 7 self-checking test programs in `tests/qbe/` (per D-09, D-10, D-11). Same pattern as Plan 01 tests: use only builtins, explicit `exit(0)`, self-checking with FAIL messages.

**test_arrays.sb** -- Tests Array type, ArrayAccess, array literal:
- Create array: `let arr: int[] = [10, 20, 30]`
- Access by index: `arr[0]` should be 10, `arr[2]` should be 30
- Array assignment: `arr[1] = 99` then verify `arr[1] == 99`
- Array length (if supported): test `len(arr)` returns 3
- Note: if array syntax differs, consult examples/bubblesort.sb for working pattern

**test_structs.sb** -- Tests StructInitialization, FieldAccess:
- Define struct: `struct Point { x: int, y: int }`
- Initialize: `let p = new Point { x: 10, y: 20 }`
- Field access: `p.x` should be 10, `p.y` should be 20
- Nested struct: `struct Rect { origin: Point, width: int }` -- init and access `r.origin.x`
- Multiple struct instances

**test_methods.sb** -- Tests Selff expression, method calls:
- Struct with method: `struct Counter { value: int }` with `fn Counter.increment(self): int { return self.value + 1 }`
- Call method and verify result
- Method accessing multiple fields

**test_for_loops.sb** -- Tests For statement (for-in, lowered from HAST):
- For-in over array: `for x in arr { ... }` -- accumulate sum and verify
- For-in over range if supported
- Nested for loops if feasible

**test_match.sb** -- Tests match statement (lowered to if-else by transformer):
- Match on integer: `match x { 1 => ..., 2 => ..., _ => ... }`
- Verify correct branch executes for each value
- Match with multiple cases

**test_compound_assign.sb** -- Tests +=, -=, *=, /= BinOps:
- `let x = 10` then `x += 5` verify `x == 15`
- `x -= 3` verify `x == 12`
- `x *= 2` verify `x == 24`
- `x /= 4` verify `x == 6`

**test_nested_expressions.sb** -- Tests complex nested BinOps and field access chains:
- Nested arithmetic: `(1 + 2) * (3 + 4)` should be 21
- Chained field access: struct with nested struct, access deeply
- BinOp with function call: `add(1, 2) + add(3, 4)` should be 10
- Mixed: comparison of computed values

Important: Some tests may fail at compile time or produce wrong results. That is EXPECTED and valuable. If a test program fails to compile, comment out the failing section and add a note: `// COMPILE_FAIL: <description>`. If it compiles but produces wrong results, the harness will catch it. Both outcomes are recorded in the gap inventory.
